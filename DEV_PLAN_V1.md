# DEV_PLAN_V1

## 项目定位

产品名称：打赌模拟器

核心场景：
- 朋友之间临时打赌记账
- 快速录入下注信息
- 在结果出来后进行结算记录
- 生成好看的图片用于保存或转发

产品目标：
- 页面录入效率高
- 导出图片好看
- 支持多赌局管理
- 结算准确
- 微信小程序体验好

## 版本策略

当前先做 `V1`，不做云端，不做服务器，不做数据库部署。

数据策略：
- 全部历史数据本地化存储
- 基于 `uni.setStorageSync` / `uni.getStorageSync`
- 优先保证单机可用、低成本、可持续迭代

明确不做：
- 登录系统
- 多人协作
- 云同步
- 自动判断事件结果
- 付费部署服务端

## 业务流程

一个赌局完整生命周期：

1. 新建
2. 加人下注
3. 关闭下注
4. 出结果
5. 结算
6. 存档

说明：
- “出结果”不由小程序自动判断
- 用户手动选择结果即可
- 小程序负责记录与结算展示

## V1 功能范围

### 1. 多赌局管理
- 支持新建赌局
- 支持切换赌局
- 支持删除未归档赌局
- 支持本地保存多个赌局

### 2. 赌局基础信息
- 标题可编辑
- 支持正方 / 反方两边下注
- 每条下注包含：
  - 姓名
  - 金额
  - 是否已缴

### 3. 实时计算
- 总奖池
- 正方赔率
- 反方赔率
- 双方小计
- 每个参与人的预期收益

### 4. 赌局状态
建议状态：
- `draft`：草稿 / 进行中
- `locked`：已关闭下注
- `settled`：已结算
- `archived`：已归档

状态说明：
- `draft`：允许继续编辑下注
- `locked`：禁止继续加人下注，但还没出结果
- `settled`：已经记录结果并给出结算金额
- `archived`：历史归档，只读展示

### 5. 出结果
结果类型建议：
- `positive`
- `negative`
- `draw`
- `pending`

说明：
- `positive`：正方赢
- `negative`：反方赢
- `draw`：流局 / 原额退回
- `pending`：尚未出结果

### 6. 结算逻辑
结算规则建议：

#### 正方赢
- 正方每个人收益 = 个人投入 × 正方赔率
- 反方全部记为输

#### 反方赢
- 反方每个人收益 = 个人投入 × 反方赔率
- 正方全部记为输

#### 流局
- 所有人原额退回

说明：
- 当前版本先做“应得金额”展示
- 不做复杂的转账流水或实付核销流程

### 7. 图片导出
导出目标：
- 能生成适合保存/转发的图片
- 适配微信小程序
- 保持视觉清晰

V1 建议支持两类图片：
- 进行中海报
- 结算结果海报

### 8. 历史本地化
- 已结算的赌局可以归档
- 能在本地查看历史赌局
- 归档后的赌局只读

## 页面规划

### 页面 1：首页 / 赌局列表页
用途：
- 查看全部赌局
- 过滤状态
- 新建赌局
- 进入赌局详情

建议模块：
- 顶部标题
- 状态筛选：进行中 / 已关闭 / 已结算 / 已归档
- 赌局卡片列表
- 新建赌局按钮

### 页面 2：赌局详情页
用途：
- 录入下注
- 查看赔率与小计
- 关闭下注
- 选择结果
- 查看结算
- 导出图片
- 归档赌局

建议模块：
- 赌局标题
- 状态标签
- 总奖池
- 正反双方录入区
- 规则说明
- 关闭下注按钮
- 结果选择区
- 结算结果区
- 导出图片按钮
- 归档按钮

### 页面 3：归档详情页（可选）
V1 可以先不单独拆页面。
如果首页列表已经能查看归档赌局详情，则可以先复用详情页只读模式。

## 数据结构建议

### 赌局对象

```js
{
  id: 'game_xxx',
  title: '明天会下雨吗?',
  status: 'draft',
  result: 'pending',
  positiveBets: [],
  negativeBets: [],
  positiveOdds: 0,
  negativeOdds: 0,
  totalPool: 0,
  lockedAt: null,
  settledAt: null,
  archivedAt: null,
  createdAt: 0,
  updatedAt: 0
}
```

### 下注对象

```js
{
  name: '小明',
  amount: 50,
  paid: true,
  settlementAmount: 0,
  settlementStatus: 'pending'
}
```

可选状态：
- `pending`
- `won`
- `lost`
- `refund`

## 推荐下一步开发顺序

### 阶段 1：状态模型升级
优先做：
- 给赌局增加 `status`
- 给赌局增加 `result`
- 给赌局增加时间字段
- 给下注增加 `settlementAmount`
- 给下注增加 `settlementStatus`

### 阶段 2：关闭下注
功能：
- 增加“关闭下注”按钮
- `draft -> locked`
- 关闭后禁止继续新增/删除/编辑下注

### 阶段 3：出结果与结算
功能：
- 手动选择：正方赢 / 反方赢 / 流局
- 自动生成结算金额
- 标记每个下注人的结算状态
- 状态变为 `settled`

### 阶段 4：归档
功能：
- 已结算赌局可归档
- 归档后只读
- 首页可查看归档列表

### 阶段 5：页面结构优化
功能：
- 从“顶部赌局切换”过渡到正式赌局列表页
- 详情页承担录入与结算
- 提升小程序可用性

### 阶段 6：图片导出升级
功能：
- 区分进行中海报与结算海报
- 结算海报展示结果与每人应得金额
- 优化视觉样式贴近设计稿

## 设计原则

开发时优先遵守：
- 小步迭代
- 单机本地优先
- 不引入不必要复杂度
- 先保证流程闭环，再打磨外观
- 数据结构为后续扩展预留空间，但不过度设计
